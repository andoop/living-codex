# OBJECTIVES — 先定目标与验收（map 的第一步）

> 大改原因：在大项目上，单 agent / 扇出失败会"浅扫几个模块就宣布完成"。先定**可验收的目标与覆盖矩阵**，AUDIT 才有对照物。

`map` 开始时，主 agent 产出 `docs/codebook/objectives.md`，含三块：

## 1. 北极星（这本书为谁、回答什么）
- 目标读者（如：接手该 Android 仓的新后端）
- 必须能回答的关键问题（如："上传链路怎么走""素材为何没传""各 flavor 差异"）

## 2. 覆盖目标矩阵（大项目必填）
先用 L1 级扫一遍**列出全部主要子系统**（module/包），再为每个标注目标深度与是否本轮必覆盖：

| 子系统 | 目标深度 | 本轮必覆盖? | 理由 |
|---|---|---|---|
| 上传/项目链路 | L3 | 是 | 用户核心关切 |
| 采集内核 | L2 | 是 | 主干 |
| 各传输实现(cos/oss/aws) | L2 | 是 | 行为差异关键 |
| 设备/算法/Flutter 等周边 | L1 清单+职责推断 | 否(下一轮) | 体量大、非本轮焦点 |

- **覆盖目标必须覆盖"全部子系统"**：哪怕只到 L1 清单级，也要列出、不能假装不存在。
- 分轮可以，但要写明**本轮目标**与**后续轮**，并在书首页公示"本轮覆盖达成度"。

## 3. 完成定义（可验收的二元标准）
- 例："① 上传链路 L3 端到端剖析成章；② 全部 ~70 子系统在模块地图有条目+职责；③ 每条'已确认'可 grep；④ AUDIT 的 P0/P1 清零。"
- SIGN-OFF 阶段逐条勾。

## 规则
- **先选覆盖模式**：`focused`（核心深读+长尾清单，可留后续轮）还是 `full`（枚举全部文件逐一画像、grind 到 100% 才算完，不留下一轮——见 `references/file-portraits.md`）。`full` 模式下覆盖矩阵退化为"全部 in-scope 文件 = 必覆盖"，objectives 主要记录排除规则与文件总数。
- **大项目 / 目标不清时**：把覆盖矩阵草案给开发者确认再开测（避免方向错、避免浅扫即完成）。
- objectives 是 AUDIT 的对照基准：**覆盖未达矩阵目标 = AUDIT 的 P1（覆盖缺口）**，驱动再做一轮。
