# references/narrative-tracing.md — 入口驱动追踪 + 双向汇合

> 落实 PRD FR2/FR3/FR6(行内标注)/FR7。验证 TC3/TC4/TC7/TC15。
> 核心铁律：**沿途调用/顺序/触发/流向一律「推断」**（行为性结论）；每条业务结论必须挂代码证据。

---

## 入口驱动追踪（FR2 · 定义"由外到里"）

从每个入口（见 `entry-ledger.md`）出发**向里追踪**控制流：把沿途每个**分支 / 判断 / 调用 / 状态变更**记为一条业务规则或子流程节点。

- 追踪是静态阅读，不是运行 → **沿途一切行为性结论最高「推断」**。
- 追踪产物：journey 草稿（节点序列）+ 证据锚点 + orphan 候选 + 挂接方式（traced/heuristic）+ **可达链**（见 `coverage-ledger.md`）。
- 跨异步/派发边界时，记录 fire 端，交主 agent 配对（见 `edge-ledger.md`），**不假装连通**。

---

## 双向汇合（FR3 · 解决"不编造"）

```
自顶向下：搭业务骨架（L0–L2 假设：这产品大概有哪些旅程）
自底向上：用代码 + 逐文件画像填充并校验
汇合规则：每条业务结论必须挂代码证据（文件 + 可 grep 符号）
          挂不上 → 删除，或降级为「未解之谜」
          骨架与证据矛盾 → 以证据为准
```

- **绝不**为了让骨架好看而脑补未挂证据的连边/结论。
- 「未解之谜」必须说明"为何没查清、该读哪个文件"（见 `usefulness-floor.md`，禁"含糊 + 推断"双免责）。

## 反例模板（#5）：字面存在(已确认) vs 调用关系(推断)

> 这是最常被违反的红线，必须随产物内置反例提醒。

❌ **错误**（把行为标已确认）：
```
UploadManager 调用 TokenStore.verify()  〔confidence: 已确认〕
```

✅ **正确**（拆成两条，区分字面存在与调用关系）：
```
- UploadManager.start 内出现对 TokenStore.verify 的调用点
  〔已确认(字面)：grep 命中 "TokenStore.verify" @ UploadManager.kt〕
- 该调用在运行期是否实际执行、被哪个实现处理（多态/DI/反射）
  〔推断：静态阅读，未运行验证〕
```

**判定准则**（继承 `provenance.md`）：凡需运行、或需读多处控制流才成立的谓语 = 行为性 → 最高「推断」。
行为性谓语包括但不限于：调用 / 在…之前 / 触发 / 回滚 / 重试 / 不重试 / 并发 / 加锁 / 校验 / 持久化 / 去重 / 路由到 / 异步回调。

## 行内/段级 confidence 标注（A1 + A-RT3-5 · 防 banner blindness）

避免"每句都标"导致 banner blindness 与零区分度，同时保留逐条可辨：

1. **段级横幅（默认基线）**：每节首声明本节业务叙事的基线档。
   - **追踪者/角色草稿**（证据层）可用技术化横幅：
     ```
     > 〔本节为**推断级**重建：以下流程/规则均基于静态阅读，未运行验证。〕
     ```
   - **发布 concept**（journey/rule）的横幅由 SYNTHESIZE/PRESENT 改**白话承载**（去工程师腔，见 `plain-language.md` §1、`provenance.md` §K）：
     ```
     > 🟡 下面这些是基于读代码的推断、没实际跑验证；正文只额外标 `已确认（字面存在）`/`未解之谜（没查清）` 两种例外。
     ```
2. **行内例外标记（只标与段基线不同者）保留**：
   - 更高档：`〔已确认(字面)：grep 命中 "<符号>"〕`
   - 更低档：`〔未解之谜：需运行确认重试次数〕`
   - 携带证据强度区分：`〔推断·强：单一明确分支〕` / `〔推断·弱：跨 3 文件拼接〕`
   - **例外线上可见（M8）**：`已确认`/`未解之谜` 例外不得因白话化只下沉折叠；`未解之谜` 可上浮该文档"还不知道什么"gap 小节。
3. **详尽 `文件:符号` 锚点移至线下证据区**（`## 证据与可信度`），按线上编号 ①②③ 对应；正文不堆叠锚点。
4. **禁止无任何标注的肯定式行为现在时句**——要么落在已声明基线段内、要么带行内例外标记。

> **白话化施加时机（M1/G1）**：去术语/白话化在 **SYNTHESIZE 与 PRESENT** 阶段施加于**发布 concept**（避免仅模板化妆、与生成端 split-brain）；**角色草稿/调和草稿属证据层，可保留技术性**，其结论进入发布 journey 时由 SYNTHESIZE/PRESENT 按 `plain-language.md` 白话化、按 book-spec 编号绑定。

❌ `系统断网时自动重试 3 次。`（裸句，无基线无标记）
✅ 段内（基线=推断）：`系统断网时自动重试 3 次〔推断；阈值见 RETRY_MAX 常量〕。`

## 与多角色追踪的衔接（FR15）

同一旅程由多个**不同质角色**子 agent 独立追踪（正常流程 / 数据·状态 / 失败·边界），各自不共享中间结论。分歧即信号 → 主 agent（reconciler 角色）调和。详见 `multi-role.md`。
