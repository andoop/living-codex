# references/narrative-model.md — 分层洋葱模型 L0–L5（业务轴）

> 落实 SKILL.md `narrate` 命令的呈现形态。对应 PRD FR5/FR6/§4，验证 TC1/TC6。
> 本机制复用**逐文件画像**作为 L5 证据叶子（见 `references/file-portraits.md`），不另起炉灶。
> 一切层级里的**行为/因果/顺序/连边结论最高「推断」**（见 `honesty-charter.md`）。

## 分层洋葱：由外到里 / 由浅到深

向下读 = 越来越深；每层**自包含可读**，渐进式展开（progressive disclosure）。产物落 `docs/codebook/narrative/`。

> **线上 / 线下贯穿全层**（见 `plain-language.md` §1）：L0–L3 主体 = **线上白话正文**（面向小白、零未解释术语、产品/交互视角）；confidence 锚点/三档图例/技术库名 = **线下证据区**（`## 证据与可信度`，按线上编号绑定）。L5 = 纯证据层。

| 层 | 名称 | 放什么（线上白话） | **不放什么（下沉线下）** | 产物文件 | confidence 上限 |
|---|---|---|---|---|---|
| **L0** | 产品全景 | 这是什么产品、给谁用、能干哪几件大事；≤1 页 | 模块名、类名、代码术语、营销形容词 | `L0-product.md` | 框架句须挂证据或标「编者摘要」 |
| **L1** | 能力域地图 | 几大能力域/业务模块及彼此关系；一张图 + 短描述 | 具体函数、文件路径、实现细节、营销词 | `L1-domains.md` | 同上 |
| **L2** | 用户旅程 / 业务流程 | 端到端走一遍："用户做 X → 系统做 Y → 得到 Z"，白话 | 分支穷举、行号、私有方法实现、**框架/库名（→证据区）** | `journeys/<journey>.md`（L2 段） | 行为结论「推断」 |
| **L3** | 场景与规则 | 每条流程展开的分支/条件/边界/异常/业务规则——**细节主体在此层**，白话且决策级 | 逐行代码、与业务无关的工具调用、**未解释术语（→证据区/glossary）** | `journeys/<journey>.md`（L3 折叠）+ `rules/` | 「推断」+ 决策级具体度（见 `usefulness-floor.md`） |
| **L4** | 机制实现 | 状态机、数据模型、关键算法、时序 | 完整源码贴片 | `journeys/<journey>.md`（L4 折叠） | 「推断」；Mermaid 带「非验证协议」caption |
| **L5** | 代码锚点 | **证据层（below the line）**：链到 `文件:符号` + 逐文件画像；**保留技术术语正常** | —（这是叶子，往下就是代码本身） | 互链 `portraits/` 或直链 `file:symbol` | 字面存在性可「已确认」、调用关系「推断」 |

> **去术语下沉 L2/L3**：原 L0/L1 的"禁模块名/类名/代码术语/营销词"纪律**扩展到 L2/L3 正文**——L2/L3 主体是白话产品/交互叙事，代码术语进 L5/证据区；术语三类处置见 `plain-language.md` §2。

## 每层「放什么 / 不放什么」判定线

- **判定问题**：读者在这一层想回答的问题是什么？答案粒度高于该问题 = 上移，低于 = 下沉折叠。
  - L0：「这软件是干嘛的？」→ 一句话产品定位 + 3–7 件大事。
  - L1：「它由哪几块能力组成、怎么协作？」→ 能力域 + 一张关系图。
  - L2：「我做一件事，端到端发生了什么？」→ 主干顺序，分支收起。
  - L3：「这一步在什么条件下会怎样？阈值/边界/异常？」→ 决策级具体度。
  - L4：「底层怎么实现的？」→ 状态/数据/时序。
  - L5：「证据在哪？」→ 代码锚点。
- **上层只给骨架与结论，细节折叠下层**（progressive disclosure）；同一事实不在两层重复展开，只在归属层展开、其余层引用。

## L0/L1 框架句纪律（PRD A7 · 禁营销/评价形容词）

- **禁止**在 L0/L1 框架句使用无证据的评价/营销词：`智能地 / 可轻松 / 高效 / 无缝 / 强大 / 安全可靠` 等。
- 框架句二选一：
  1. **挂证据**：`本产品支持离线拍摄〔证据：manifest 注册的 OfflineCaptureService :: ...〕`；或
  2. **显式标「编者摘要（非结论）」**：`> 编者摘要（非结论）：以下为编者基于阅读的概括，非可追溯结论。`
- 评价类断言（"性能好""体验流畅"）需运行/度量才成立 → **一律不写**或进「未解之谜」。

## 逐节 confidence 图例（三档纪律，见 `provenance.md`）

每个 journey/rule 文件章首放**一句白话可信度声明**（承载"推断"默认基线，D5），例：
> 🟡 下面这些是基于读代码的推断、没实际跑验证；正文只额外标 `已确认（字面存在）` 与 `未解之谜（没查清）` 两种例外。

- **完整三档释义图例下沉**到该文档 `## 证据与可信度`（线下）或链 `glossary.md`（见 `plain-language.md` §1、`provenance.md` §K）——避免正文行内堆叠术语噪声。
- **例外仍线上可见（M8）**：`已确认`/`未解之谜` 例外行内保留或上浮 gap 小节，不得只下沉折叠。
- confidence 标注**归线下**，正文用上面章首一句白话声明承载基线。

详细诚实口径见 `honesty-charter.md`（一份宪章统管三处免责 + 机械闸门 vs AUDIT 能/不能对照表）。
