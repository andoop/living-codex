# references/multi-role.md — 多角色协作交叉验证（FR15）

> 落实 PRD FR15（开发者确认原则 · 部分缓解 A-RT3-4 bias）。验证 TC25。
> 核心：**禁止单个 agent 自始至终独自决定业务叙事。**

---

## 多角色不同质追踪

同一条业务旅程由**多个不同角色种子的子 agent 独立追踪**，再交叉调和。角色不同质（落到追踪种子）：

| 角色 | 种子 | prompt |
|---|---|---|
| 「正常流程」追踪者 | 主干 happy path 入口 | `prompts/narrative-trace-role-normal.md` |
| 「数据·状态」追踪者 | 状态/条件触发种子（配合 FR13） | `prompts/narrative-trace-role-state.md` |
| 「失败·边界·异常」追踪者 | 异常分支、错误处理、边界 | `prompts/narrative-trace-role-failure.md` |

各自独立从不同入口/状态种子出发，**不共享中间结论**。

---

## 分歧即信号

多角色叙事出现**分歧/冲突**处 = 业务被误读或本就含糊的**强信号**：
- 主 agent（**reconciler 角色，非任一追踪者**）核实调和（`prompts/reconcile.md`）。
- 无法调和的 → **降「未解之谜」并显式标注分歧**，**不靠多数表决假装定论**。

## 流水线角色分离

```
追踪者  ≠  综合者  ≠  红蓝审核者  ≠  调和者
```
任一环节不由同一 agent 独揽端到端。编排仍遵 FR9 集中化：**仅主 agent 写共享状态/账本**。

> **白话化职责归属**：追踪者/调和者草稿 = 线下证据层，可保留技术性；**白话化（去术语、线上/线下分层、编号绑定）由 SYNTHESIZE/PRESENT 在归并发布时施加**（见 `plain-language.md`、`narrative-tracing.md`），不在追踪/调和环节施加，避免污染独立性与证据保真。

---

## 诚实边界（防在新层冒充权威）

- 多角色**降低相关性偏差、提升而非保证客观**——所有子 agent 仍**同模型类、读同一静态源、无运行时/产品 oracle**，**共享的系统性误读仍可能一致漏过**。
- 故多角色交叉**不升级** §0.5.1：**SIGN-OFF 仍不认证业务正确**，只多一句"经多角色交叉、分歧已暴露/调和"。

## 串行降级 + 戳记披露（RT4-3 · 防戳记冒充未交付的交叉）

宿主并行不可用时降级为**串行多角色**（同一主 agent 切换显式角色 + 独立上下文轮次），**不得退化为单角色一遍过**。

但串行模式下同一 agent 切角色**无法真隔离上下文**、"分歧即信号"显著受限。故 SIGN-OFF 戳记与首页**必须区分并行/串行**：

- 并行：`经多角色独立交叉验证（并行）`
- 串行：`⚠️ 本轮为串行单 agent 多角色，独立性与分歧检测能力显著受限，不等同独立交叉验证`

**禁止**串行产物盖与并行同款的"经多角色独立交叉"戳记。
