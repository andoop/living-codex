# 监督与签收闭环 · AUDIT + SIGN-OFF

> **理解的产物必须被独立审核，不能由写书的 AI 自己宣布完成。** 这是本 skill 对"AI 说做完就做完"的根治。

## 为什么需要（来自真实验收反馈）
在 13 万文件的大型 Android 仓上，并行扇出失败 → 单 agent 串行只深读了 7/70 模块 → 却走到"完成"。诚实标注虽在，但**没有任何独立环节对照目标揪出"覆盖严重不足"**。AUDIT 就是补这一环。

## AUDIT — 独立子 agent 对抗式审核（PRESENT 后必做）

主 agent 扇出**至少两路独立审核子 agent**（**必须与写书 agent 不同上下文**；宿主无并行则串行依次起独立 agent / 子进程，**不得**用写书上下文自审顶替）：

### A. 头脑预演子 agent（覆盖 / 逻辑闭环）
读 `objectives.md` + 全书，回答：
- 覆盖目标矩阵里**每个"本轮必覆盖"子系统**是否真有对应深度的章节？漏哪些？
- 章节间是否一致（同一事实多处说法是否冲突）？
- "部分测绘 / 未解之谜" 是否如实标注，还是把"没读"伪装成"读过"？
- 返回 `LOGIC_CLOSED`（覆盖达标+闭环）或 `ANOMALY_FOUND`（含 P0–P3）。

### B. 红蓝对抗子 agent（攻击这本书）
攻击向量：
- **过度声明**：把行为/关系/运行期结论标了「已确认」？
- **编造/失效引用**：抽样「已确认」跑 `verify-citations.sh`，命中率？有没有指向不存在符号？
- **覆盖缺口**：objectives 要求覆盖却漏测/浅测的关键子系统（按用户关切排序，不是凑数）。
- **完成度造假**：覆盖远未达目标却口吻像"已完成 / 全面"。
- **密钥泄漏**：产物/中间产物有无密钥/PII 原文。
- 每记杀招给可复现证据 + P0–P3。返回 `HELD` 或 `BREACH_FOUND`。

审核报告写入 `docs/codebook/audit/audit-<n>.md`（主 agent 亲核，不轻信审核 agent，抽查其证据）。

## SIGN-OFF — 对照目标验收

```
P0/P1 清零?  且  覆盖达 objectives 目标(或开发者明确接受残余)?
   ├─ 是 → 标 DONE：书首页写"对照 objectives 覆盖达成度 + 残余 + 审核轮数"
   └─ 否 → 回 SURVEY/SYNTHESIZE/PRESENT 再做一轮(深化漏测/修过度声明) → 再 AUDIT → 循环
```

- **覆盖缺口（漏测 objectives 必覆盖项）= P1**，驱动再做一轮（除非开发者明确接受降级/分轮）。
- **再做一轮的范围聚焦**：只补漏测/被攻破处，不重测已达标 territory（按 survey-state 续跑）。

## 与 review 的区别（别混用）
| | `review`（自审） | `AUDIT`（监督） |
|---|---|---|
| 谁做 | 写书的同一上下文 | **独立**子 agent |
| 查什么 | 确定性项（失效引用/wikilink/孤儿） | 覆盖达标 + 对抗式找漏洞/过度声明 |
| 能否判完成 | 否 | **是（SIGN-OFF 的依据）** |

## 红旗
- 扇出失败 → 浅扫即宣布完成（**最严重**，正是本次反馈）。
- 用 `review` 顶替 `AUDIT`。
- 审核子 agent 与写书 agent 同上下文（回声，不算独立审核）。
- 覆盖 10% 却不在书首页标"目标缺口"。
