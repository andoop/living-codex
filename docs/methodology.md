# 方法论 · How It Works

Living Codex 就是一份 `SKILL.md`：AI 加载后，按下面的流程把代码库讲成一本业务书。完整规范在 [`skill/SKILL.md`](../skill/SKILL.md) 与 `skill/references/`。

## 一条旅程是怎么攒出来的

| 步骤 | 在干嘛 |
|---|---|
| **① 定范围** | 排除构建产物、依赖、二进制等，列出一份干净的文件清单当"覆盖底数"。 |
| **② 列入口** | 从框架的路由、注解、配置（Spring 注解、SPI、依赖注入、事件总线、AndroidManifest…）里**独立**列出所有业务入口，每个要么"算一条业务"，要么"明确排除"。再补一轮状态触发器（登录过期、缓存失效、权限、灰度开关、数据迁移…），堵住那些不是用户点出来的业务。 |
| **③ 多角色追踪** | 同一条旅程派三个视角各自独立追：正常流程 / 数据与状态 / 失败与边界。三者互不通气，只读代码、不改代码。 |
| **④ 配连边** | 旅程常被异步、消息、派发切成几段。把这些断点重新接上；接不上的老实标"断裂边"，不假装连通。 |
| **⑤ 调和** | 三个视角打架的地方，由专门的"调和者"核实。定不下来就降级成"未解之谜"，不靠少数服从多数硬定。 |
| **⑥ 成书** | 归并成 L0–L5 六层洋葱，写出产物（首页 + 各旅程 + 规则 + 代码证据 + 各账本 + 盲区专章 + Mermaid 图）。 |
| **⑦ 审 + 验收** | 另派一个独立 agent 挑刺（红蓝对抗）；再跑机械校验脚本对账。**脚本通过、问题清零，才算完成**——不是 AI 自己说做完就完。 |

## 六层洋葱 L0–L5

由外到里、由浅到深，每层都能单独读：

| 层 | 回答的问题 |
|---|---|
| **L0 产品全景** | 这软件是干嘛的？给谁用？能干哪几件大事？ |
| **L1 能力域** | 它由哪几块能力组成、怎么协作？ |
| **L2 用户旅程** | 我做一件事，端到端发生了什么？ |
| **L3 场景与规则** | 这一步在什么条件下会怎样？阈值、边界、异常？ |
| **L4 机制实现** | 底层的状态、数据、时序怎么实现的？ |
| **L5 代码锚点** | 证据在哪？（链到具体 `文件:符号` 和逐文件画像） |

## 命令一览

- **`narrate [path]`** — 梳理全项目业务旅程。
- **`narrate --journey <名>`** — 只剥一条旅程（首次推荐，跑得快）。
- **`narrate --resume`** — 中断后接着跑。
- **`narrate --audit`** — 单独再跑一轮独立审核。
- **`narrate --lint`** — 代码改后做增量保鲜，标记可能过期的旅程（不阻断）。

## 关于可选后端

`--backend codegraph|repomix|none`。后端是**锦上添花**：装了就用来让静态结论更可信、或加速读取；没装就降级运行，结构不缺、承诺同步降低。**核心永远零运行时、可移植。**
